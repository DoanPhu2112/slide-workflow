Vấn đề trong phương án SSL hiện tại

Current SSL

  • Ưu điểm: inspect host, path, header và áp policy Layer 7
  • Ưu điểm: offload TLS cho backend
  • Đổi lại: nếu không re-encrypt, trust chuyển vào mạng nội bộ
  • Inspect chỉ xuất hiện sau khi payload được giải mã

Client → HTTPS → Edge proxy → HTTP hoặc re-encrypt → App

Các phương án SSL

SSL passthrough

  • TLS đi xuyên qua proxy
  • Backend mới là nơi giải mã
  • Proxy nhẹ hơn
  • Không inspect được HTTP

SSL bridge

  • Proxy giải mã để inspect
  • Proxy tạo TLS session mới tới backend
  • Có thể inspect trong quá trình xử lý trong Proxy

SSL terminate

  • Proxy giải mã ở edge
  • Edge route dựa trên HTTP request thật
  • L7 control mạnh nhất
  • Đơn giản hơn bridge

SSL Tradeoffs

Tiêu chí Passthrough Bridge Terminate

Bridge và terminate inspect mạnh hơn. Passthrough và bridge giữ internal hop kín hơn.

Cách SSL hoạt động

TLS Handshake

Handshake dùng để làm gì

  1. Trao đổi message để hai bên ACK và mở đầu phiên.
  2. Xác thực server bằng certificate; có thể xác thực hai chiều nếu dùng mTLS.
  3. Đồng thuận version TLS, cipher suite và tham số crypto sẽ dùng.
  4. Sinh ra session key để bảo vệ application data phía sau.

Sau handshake có gì

  • Client biết đang nói chuyện với ai.
  • Hai bên có chung session key nhưng không truyền plaintext key trên wire.
  • HTTP data chỉ bắt đầu được mã hóa sau khi handshake hoàn tất.
Handshake diễn ra trước. Chỉ sau khi có session key thì request và response HTTP mới đi dưới dạng encrypted application data.

Certificate & Trust

Proxy đăng ký certificate với CA

  1. Tạo private key.
  2. Tạo CSR.
  3. Gửi CSR cho CA.
  4. CA xác minh domain hoặc tổ chức.
  5. CA cấp certificate.
  6. Cài certificate và private key lên F5, NGINX hoặc Ingress.

Client tin certificate dựa vào đâu

  • Browser, OS hoặc runtime sẽ có sẵn danh sách root CA đáng tin.
  • Nếu certificate chain nối được về một root CA trong trust store thì certificate được chấp nhận.
  • Private hoặc internal root CA chỉ được tin khi bạn chủ động import vào trust store.

Thuật ngữ

Edge Security Terms

WAF

Web Application Firewall.

Lọc request HTTP/HTTPS để chặn pattern tấn công như SQLi, XSS, path traversal.

Trong ARP: nằm ở F5 để chặn traffic xấu trước khi vào NGINX.

SNI

Server Name Indication trong TLS handshake.

Cho client nói trước hostname muốn truy cập dù payload HTTP chưa được giải mã.

Trong passthrough: proxy chỉ có thể đọc SNI để route TLS stream.

JWT

JSON Web Token.

Token mang claim định danh và quyền, thường được verify bằng signature.

Trong ARP: Ingress / Istio dùng JWT để chỉ cho request hợp lệ đi tiếp.

Service-to-Service Terms

mTLS

Mutual TLS.

Cả client và server đều xuất trình certificate để xác thực hai chiều.

Trong service mesh: bảo vệ east-west traffic giữa các service.

HMAC

Hash-based Message Authentication Code.

Dùng shared secret để chứng minh message không bị sửa và đúng người gửi.

Trong backend integration: phù hợp cho callback, webhook, request signing.

Vault

Secret management system.

Lưu và cấp phát secret, token, certificate theo policy thay vì hardcode trong app.

Trong ARP: dùng để giữ credential và rotate secret an toàn hơn.

Data Protection Terms

SSE-KMS

Server-Side Encryption with KMS-managed keys.

Dữ liệu được mã hóa ở storage side, còn key được quản lý bởi Key Management Service.

Phù hợp khi muốn kiểm soát quyền dùng key và audit key usage.

Encryption at Rest

Mã hóa dữ liệu khi đang được lưu trữ.

Khác với TLS là bảo vệ khi truyền, khái niệm này bảo vệ disk, DB, object storage.

Trong ARP: áp dụng cho backend storage và dữ liệu nhạy cảm.

Basic Auth

Cơ chế gửi username/password trong header Authorization.

Đơn giản nhưng phải đi kèm TLS vì credential chỉ được base64, không tự mã hóa.

Trong backend services: phù hợp cho một số hop nội bộ hoặc integration đơn giản.
